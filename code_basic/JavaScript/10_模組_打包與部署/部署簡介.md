# 部署簡介 (Deployment)

**部署 (Deployment)** 是將開發完成的網站或應用程式發布到伺服器上，使其可以被全球的使用者存取。對於前端應用程式而言，部署通常涉及將靜態資源（HTML, CSS, JavaScript, 圖片等）放置在 Web 伺服器或內容分發網路 (CDN) 上。

## 1. 部署流程概覽

一個典型的部署流程可能包括以下步驟：

1.  **程式碼審查與測試**：確保程式碼品質良好，所有功能正常運作。
2.  **建構 (Build)**：將開發環境的程式碼轉換為生產環境可用的優化程式碼。這通常由打包工具 (如 Webpack, Vite) 完成，包括：
    *   程式碼壓縮 (Minification)
    *   Tree Shaking (移除未使用的程式碼)
    *   JS/CSS 檔案合併
    *   圖片優化
    *   瀏覽器相容性轉譯 (Transpilation)
3.  **上傳**：將建構好的靜態檔案上傳到伺服器。
4.  **配置**：設定伺服器環境（如 Nginx、Apache、Node.js 服務）。
5.  **測試**：在生產環境中進行最終測試，確保一切正常。
6.  **監控**：持續監控應用程式的效能和錯誤。

## 2. 常見的部署方式與平台

### (1) 靜態網站託管 (Static Site Hosting)

最適合純前端（HTML, CSS, JS）應用程式。這些平台通常提供 CDN，能夠在全球範圍內快速響應。

*   **GitHub Pages**: 免費、快速，直接從 GitHub 儲存庫部署。
*   **Vercel**: 專為前端框架優化，提供無伺服器功能，HMR 快速。
*   **Netlify**: 類似 Vercel，提供自動化部署、CDN、自訂域名等。
*   **Firebase Hosting**: Google 提供的服務，整合其他 Firebase 功能。
*   **Amazon S3 / CloudFront**: S3 用於儲存靜態檔案，CloudFront 提供 CDN 加速。

**部署步驟範例 (以 Vercel 為例):**

1.  `npm run build` (或 `yarn build`)：產生 `dist` (或 `build`) 資料夾。
2.  安裝 Vercel CLI：`npm install -g vercel`
3.  在專案根目錄執行：`vercel`
4.  依照指示操作，Vercel 會自動偵測框架、執行建構並部署。

### (2) 傳統 Web 伺服器

將靜態檔案手動上傳到傳統的 Web 伺服器。

*   **Nginx / Apache**: 設定伺服器指向你的靜態檔案目錄。

```nginx
# Nginx 配置範例
server {
    listen 80;
    server_name yourdomain.com;
    root /var/www/your_app/dist; # 你的應用程式靜態檔案路徑
    index index.html;

    location / {
        try_files $uri $uri/ /index.html; # 用於 SPA 路由
    }
}
```

### (3) 雲端服務供應商 (Cloud Providers)

*   **AWS (Amazon Web Services)**: S3, CloudFront, EC2, Elastic Beanstalk
*   **Google Cloud Platform (GCP)**: Firebase, Compute Engine, App Engine
*   **Microsoft Azure**: Azure Static Web Apps, App Service

這些平台提供豐富的服務，從靜態託管到完整的伺服器架構。

## 3. 持續整合/持續部署 (CI/CD)

為了自動化部署流程，現代開發通常會整合 CI/CD 工具。

*   **GitHub Actions**: 整合在 GitHub 儲存庫中，監聽 Push 事件，自動執行測試和部署。
*   **GitLab CI/CD**: 整合在 GitLab 中。
*   **Jenkins**: 較為傳統的開源 CI/CD 工具。

透過 CI/CD，每次程式碼合併到主分支後，測試、建構和部署都可以自動完成，大大提高了開發效率和部署頻率。
