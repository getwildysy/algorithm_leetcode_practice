# 【試寫樣章：1.3 React Server Components 解密】

## 標題：廚房與外場：重新思考元件的分工

想像一下，你經營著一家高級餐廳（這就是你的 Web App）。

在過去的 React (SPA) 時代，我們習慣把所有的食材（程式碼）、鍋具（邏輯）甚至廚師（運算能力）都一股腦地塞進大卡車，運到客人的餐桌旁（瀏覽器），然後在客人面前現場切菜、烹飪。這聽起來很酷，但如果你的菜單很複雜，客人就得餓著肚子等你的卡車卸貨、等廚師準備，這就是為什麼你的網頁載入會慢（Bundle Size 太大）。

Next.js 的 App Router 引入了 Server Components，徹底改變了這個流程。現在，我們把元件分成了兩類：

### 1. Server Components (後場廚房)：
這是預設狀態。所有的備料、繁重的運算、或是去倉庫（資料庫）拿食材的動作，統統在後場廚房完成。我們只把做好的精美菜餚（渲染好的 HTML）端給客人。
* **優點**：客人的桌子（瀏覽器）很乾淨，不需要下載那些沉重的切菜機具（npm 套件），網頁載入飛快。
* **限制**：因為是在後場，廚師聽不到客人的即時反饋（不能使用 onClick, useState, useEffect）。

### 2. Client Components (前場服務生)：
當你需要和客人互動時，比如客人要點餐、要填寫問卷、或是點擊一個「讚」的按鈕，這時候就需要派出 Client Component。
* **如何標記**：你只需要在檔案最上方加上一行咒語：`"use client"`。這就像是告訴 Next.js：「嘿，這個元件要派到外場去，請幫我打包相關的 JavaScript 到瀏覽器。」

### 結論：
不要把所有邏輯都推給瀏覽器。讓伺服器做伺服器擅長的事（讀資料库、運算），讓瀏覽器做瀏覽器擅長的事（互動、動畫）。這就是現代 Next.js 開發的核心心法。
