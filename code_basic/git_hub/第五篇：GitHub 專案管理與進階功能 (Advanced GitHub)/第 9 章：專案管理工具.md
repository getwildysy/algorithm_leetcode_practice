# 第 9 章：專案管理工具

GitHub 不僅僅是程式碼儲存庫，它本身就是一套強大的專案管理系統。本章介紹如何利用 GitHub 來管理開發進度。

## 9.1 Issues：追蹤 Bug 與新功能建議

**Issues** 是 GitHub 專案的討論區，也是任務追蹤的核心。

### 良好的 Issue 結構 (Markdown)
一個好的 Issue 應該包含結構化的資訊。你可以使用 **Issue Templates** 來規範大家的回報格式：

```markdown
<!-- .github/ISSUE_TEMPLATE/bug_report.md -->

## 🐛 Bug 描述
請簡述發生了什麼錯。

## 📸 重現步驟
1. 前往 '...'
2. 點擊 '....'
3. 看到錯誤訊息 '....'

## 💻 環境資訊
- OS: [e.g. Windows 10]
- Browser: [e.g. Chrome 90]
```

### 任務清單 (Task Lists)
在 Issue 描述中，你可以使用這種語法來建立 checkbox：

```markdown
- [x] 完成前端切版
- [ ] 串接登入 API
- [ ] 撰寫單元測試
```
GitHub 會自動把它渲染成可以勾選的清單，並在列表頁顯示完成進度 (e.g., "1 of 3 tasks").

## 9.2 Labels 與 Milestones 的應用

### Labels (標籤系統)
這就像是便利貼的顏色分類。

| 標籤 | 顏色 | 意義 |
| :--- | :--- | :--- |
| `bug` | 🔴 紅 | 程式錯誤，優先處理 |
| `enhancement` | 🔵 藍 | 新功能建議 |
| `documentation` | 🟡 黃 | 只是文件修改，不涉及程式碼 |
| `wontfix` | ⚪ 白 | 經討論後決定不修 |
| **`good first issue`** | 🟣 紫 | **新手友善** (常用來招募新貢獻者) |

### Milestones (里程碑)
將一群 Issues 綁定到一個目標 (Target)。
*   **範例**：`v2.0 Beta Release` (截止日：12/31)
*   **功能**：你可以看到「燃盡圖 (Burndown Chart)」，知道離目標還差多少 Issue 沒關。

## 9.3 GitHub Projects (Kanban Board)

GitHub 內建了類似 Trello / Jira 的看板功能。

```mermaid
kanban
  Todo
    [ ] 設計資料庫 Schema
    [ ] 購買網域
  In Progress
    [ ] 實作 Login API @backend-team
    [ ] 切版首頁 @frontend-team
  In Review
    [ ] PR #42: User Profile
  Done
    [x] 環境建置
```

**自動化流程 (Automation)**：
你可以設定規則，例如：
*   當 Issue 被建立時 -> 自動跑進 **Todo** 欄位。
*   當 PR 被 Merge 時 -> 自動把關聯的 Issue 移到 **Done** 欄位。

## 9.4 Wiki 與 README.md

### README.md：專案的門面
這是別人進入你專案首頁第一眼看到的檔案。

#### 必備章節 (Checklist)
1.  **Project Title & Banner**: 讓人知道這是什麼。
2.  **Getting Started**: 如何在 5 分鐘內讓專案跑起來。
3.  **Prerequisites**: 需要安裝 Node.js? Python? Docker?
4.  **License**: MIT? Apache? (這決定了別人敢不敢用你的 code)。

### Wiki：知識庫
適合存放：
*   API 文件
*   架構設計圖
*   開發者手冊 (Contribution Guide)
